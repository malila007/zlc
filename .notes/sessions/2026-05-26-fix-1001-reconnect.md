# Session: Fix WebSocket 1001 Reconnect Bug

**Date:** 2026-05-26

## Goal
แก้ production bug: หลัง server restart, tab ทั้งหมดไม่ reconnect

## Root Cause
Server ส่ง WS close code 1001 ตอน graceful shutdown (PM2/Docker restart)
Client code ใน `onclose` treat 1001 เป็น "intentional client disconnect" → ไม่ reconnect
Client's own `disconnect()` ลบ `onclose` listener ก่อน close() เสมอ ดังนั้น 1001 จาก client จะไม่มีทางถึง onclose ได้ — safe to remove the guard

## What Changed
- `connection-manager.js` `onclose` handler: ลบ `isManualDisconnect` (code 1001) guard ออก
- code 1000 = clean close → `resetReconnection()` (no reconnect) — unchanged
- code 1001, 1006, 4000, 4001, etc. → `onConnectionLost()` → reconnect — fixed
- heartbeat timeout: ใช้ `ws.close(4000, "Heartbeat timeout")` ชัดเจน ไม่พึ่ง browser default

## Second Fix: FORCE_RECONNECT Loop (refresh leader tab scenario)
**Symptom**: เปิด 2 tabs, refresh leader → ทั้งคู่ค้าง "Connecting..." ตลอด

**Root cause**:
1. Tab A refresh → Tab B (เคยเป็น follower) ได้ lock → กลายเป็น leader → เริ่ม connect WS
2. Tab A new page mount เป็น follower → offline banner ขึ้น (chat open + isConnected=false)
3. Banner's `runAutoRefreshThen` ยิง `chatService.refreshChat()` ทันที (immediate, ไม่ delay)
4. → ส่ง `FORCE_RECONNECT` ผ่าน BroadcastChannel ไป Tab B
5. Tab B handler: `this.forceReconnect(this.agentId)` → **ฉีก WS ที่กำลัง auth ทิ้ง** เริ่มใหม่
6. Banner ติ๊กต่อ → ยิง `refreshChat` อีก → ฉีก WS อีก → **loop ไม่จบ**
7. Memory pressure ทำให้ MongoDB auth ช้า ยิ่งทำให้ loop run นานขึ้น

**Fix**: `becomeLeader()` FORCE_RECONNECT handler ตรวจ `ws.readyState` ก่อน:
- ถ้า CONNECTING/OPEN → ถ้า `state.isConnected=true` ก็แค่ re-broadcast `WS_CONNECTED` (sync follower's UI); ถ้ายัง auth อยู่ก็ปล่อยให้ทำต่อ
- ถ้า ws null/CLOSED → reconnect ตามปกติ

## Connection to Memory Swap Session
Memory limits จาก session `2026-05-26-memory-swap-fix.md` ทำให้ container อาจถูก restart บ่อยขึ้น
ทุกครั้งที่ PM2/Docker ส่ง SIGTERM → graceful shutdown → 1001 → client stuck
